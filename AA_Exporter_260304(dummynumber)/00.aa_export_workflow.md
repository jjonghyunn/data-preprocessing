기준 문서 업데이트일: `260316`

---

안녕하세요. 팀, 박종현입니다.

NYNY 캠페인 약 2시간 정도로 정제까지 다 되었고 검수상 제출한 엑셀 raw와 같았습니다.
(26, 26-prior, 25년 기간 전 site 추출 기준이며, 미추출 건 수기 보정 등 변수에 따라 소요 시간이 달라질 수 있습니다.)

아래는 AA Export Workflow입니다. STEP 1, 2, 3은 준비 단계이고 STEP 5부터가 실제 API 데이터 추출 단계입니다.

---

## 권장 실행 순서

1. 마스터 CSV 최신화 (sitecode, tb_column_name_mapping, currency)
2. JSON 생성 및 세그 검수 (json폴더 내 `empty_json_maker_by_input_tb_name.py`, `json_segment_checker.py` 등 활용)
3. `generate_period_notebooks_v2.py` 실행
4. 노트북 6개 실행 (3가지 기간 global/us) (노트북 자동 생성 단계)
5. `check_failed_status.py` 실행 후 수기 보정
6. `check_mapping_match.py` 실행
7. `stack_n_currency_n_chnl_n_seaprate_날짜.ipynb` 실행 (데이터 정제/제출용 엑셀 raw 포맷팅 단계)
8. `aa_exports/union_{timestamp}.csv` 최종본 확인

실제 기준 스크립트는 아래 4개입니다.

- `generate_period_notebooks_v2.py`
- `ipynb_json_usage_mapper.py`
- `check_mapping_match.py`
- `stack_n_currency_n_chnl_n_seaprate_날짜.ipynb`

## STEP 2. JSON Preparation

### 2-1. AA Workspace에서 1차 JSON 추출

- 기준은 당해 캠페인 JSON입니다.
- prior 기간과 세그 구성이 겹치는 테이블은 `26_ny_*_prior.json`으로 먼저 저장합니다.
- US는 별도 Report Suite를 쓰므로 NY와 분리해서 저장합니다.

### 2-2. prior JSON을 기준으로 non-prior / last campaign JSON 복사

`json/copy_prior_json.py` (json 폴더 내 copy_prior_json.py 파일)

- `26_ny_*_prior.json`을 `26_ny_*.json`으로 복사합니다.
- prior와 non-prior 구조가 같은 테이블의 초안 생성용입니다.

`json/copy_last_campaign_json.py`

- `26_ny_*.json`을 `25_ny_*.json`으로 복사합니다.
- 스크립트 상단 `FROM_YEAR = 26`을 기준으로 동작하므로 연도 전환 시 먼저 수정해야 합니다.
- 복사 후 `_cmp_`, `campaign`, `bestselling` 같은 캠페인 종속 세그는 25년용 세그 ID로 수동 교체가 필요합니다.

### 2-3. 빈 JSON 플레이스홀더 생성

`json/empty_json_maker_by_input_tb_name.py`

- prior에서 복사할 수 없는 테이블용 빈 JSON 파일을 일괄 생성합니다.
- 생성 직후에는 내용이 비어 있으므로 이후 AA에서 payload를 채워 넣어야 합니다.

### 2-4. AA Workspace에서 2차 JSON 마무리

- 비어 있는 JSON 파일을 차례로 채워 전체 테이블을 완료합니다.
- 추출 완료된 파일은 `-EMPTY` 접미사를 제거하면서 작업 상태를 정리합니다.

### 2-5. 미완성 JSON 표시

`json/mark_empty_json.py`

- 내용이 비어 있는 JSON에 `-EMPTY` 접미사를 다시 부여합니다.
- 최종 상태에서는 `-EMPTY` 파일이 0개가 되는 것이 정상입니다.
- 일괄 이름 복구가 필요하면 `json/rename_empty.py`를 사용합니다.

### 2-6. JSON 세그먼트 검수

`json/json_segment_checker.py`

결과 파일:

- `json_segment_report/_prior_check.csv`
- `json_segment_report/_26vs25_diff.csv`
- `json_segment_report/_filename_panel_check.csv`

검수 기준은 3가지입니다.

1. `26_ny_X.json` vs `26_ny_X_prior.json`
   - prior와 완전히 같아야 정상인 테이블 확인용입니다.

2. `26_ny_X.json` vs `25_ny_X.json`
   - `SHOULD_SAME`: 같아야 정상
   - `SHOULD_DIFFER`: 달라야 정상
   - 그 외: 참고용 표시

3. 파일명 키워드 vs panelName 키워드 일관성
   - 파일명에 `cmp` / `campaign` → panelName에 `"Campaign"` 포함 여부
   - 파일명에 `scom` → panelName에 `"S.com"` 포함 여부
   - 파일명에 `prior` → panelName에 `"Prior"` 포함 여부
   - 결과: `정상` / `확인필요` / `해당없음` / `⚠️ panelName없음`

현재 문서 기준으로 특히 확인할 포인트는 아래입니다.

- 같아야 정상: `_scom_`, `6_1_`, `6_2_`, `4_2_`
- 달라야 정상: `_cmp_`, `campaign`, `3_1_`, `4_1_`, `bestselling`

이상 없으면 STEP 3으로 진행합니다.

## STEP 3. Notebook Generation

실행 스크립트: `generate_period_notebooks_v2.py`

`json/` 폴더의 JSON 파일을 자동 감지해서 아래 6개 노트북을 생성합니다.

| 노트북 | 기간 | data_csv | JSON 패턴 |
|--------|------|----------|-----------|
| `01.26ny_campaign_period.ipynb` | 26 NY campaign | `26_ny_site_code.csv` | `26_ny_*.json` |
| `02.25ny_last_campaign_period.ipynb` | 25 NY last campaign | `25_bl_site_code.csv` | `25_ny_*.json` |
| `03.26ny_prior_period.ipynb` | 26 NY prior | `26_ny_site_code_prior.csv` | `26_ny_*_prior.json` |
| `04.us_26ny_campaign_period.ipynb` | 26 US campaign | `us_26_ny_site_code.csv` | `us_26_*.json` |
| `05.us_25ny_last_campaign_period.ipynb` | 25 US last campaign | `us_25_bl_site_code.csv` | `us_25_*.json` |
| `06.us_26ny_prior_period.ipynb` | 26 US prior | `us_26_ny_site_code_prior.csv` | `us_26_*_prior.json` |

실무 메모:

- 출력 파일명은 `OUT_NAMES`에서 바꿀 수 있습니다.
- 섹션 구분은 파일명 번호를 기준으로 자동 구성됩니다.
- `raw_multi_purchase`, `bestselling`, `nextpage`, `cc_order`, `cc_revenue`, `cc_visit`는 `base=""`로 처리됩니다.
- 각 코드 셀은 `utils.aa_exporter_jh2`를 사용하며 기본값은 `parallel_sites=True`, `max_workers=6`입니다.
- 후처리용 마지막 2개 셀은 `notebook_jh5-campaign_period.ipynb`에서 복사하도록 되어 있습니다. 해당 파일이 없으면 placeholder 셀이 들어갑니다.

## STEP 5. AA Data Extraction

생성된 6개 노트북을 순서대로 실행합니다.

- 출력 파일 형식: `aa_exports/{tb_key}_{YYYYMMDD}_{HHMMSS}.csv`
  예: `26_ny_1_1_scom_da_traffic_app_20날짜_154315`
- 기본 컬럼 구조:
  `Subsidiary, Country, Site_Code, RSID, Start_Date, End_Date, itemId, value, value1, value2, ..., status, error`
- 병렬 추출: `parallel_sites=True`
- 기본 워커 수: `max_workers=6`
- 네트워크 오류 대응: 429, 5xx 계열 재시도 로직 포함

### 추출 후 FAILED 점검

실행 스크립트: `check_failed_status_260313.py`

- `aa_exports` 루트의 원본 CSV만 검사합니다.
- `union*`, `*_stacked*`, `separate.csv`는 제외합니다.
- `status=FAILED` 건수가 있는 파일만 보여줍니다.
- US 파일은 FAILED 여부 외 **OK 행이 0개인 경우도 별도 경고** (`⚠️ 추출 값 없음`)

운영 기준:

- VRS, 일부 국가 RSID, 타임아웃 이슈는 수기 보정 대상이 될 수 있으며, 재추출을 통한 보정도 가능합니다.
- STEP 7 전에는 필요한 값이 모두 채워졌는지 반드시 확인합니다.

## STEP 6. Column Mapping Verification

실행 스크립트: `check_mapping_match_260313.py`

결과 파일:

- `mapping_match_report/_all_check.csv`
  - 출력 컬럼: `tb_key / file / status / matched_cnt / mapping_only_cnt / csv_only_cnt / matched / mapping_only / csv_only`

검수 기준:

- 원본 CSV의 `value1, value2, ...` 컬럼과 `tb_column_name_mapping.csv`의 `value_n` 컬럼이 맞는지 확인
- 기본적으로 `aa_exports` 루트만 검사
- 날짜 보관 하위폴더(`260224`, `260226` 등)는 `SEARCH_SUBDIRS = False`라서 제외
- `union*`, `_stacked*`, `_long*`, `_`로 시작하는 파일도 제외

상태 해석 (260313 기준, 6가지):

| 상태 | 의미 | 조치 |
|------|------|------|
| ✅ `정상` | 매핑과 CSV가 완전히 일치 | 그대로 진행 |
| ⚠️ `매핑초과(api 추출된 CSV부족)` | 매핑 value_n이 CSV보다 많음 (CSV에 해당 컬럼 없음) | 매핑 행 제거 또는 확인 |
| ⚠️ `CSV초과(tb_column_name매핑누락)` | CSV에 value19~20+ 등이 있는데 매핑에 없음 | 매핑에 추가 필요 |
| ⚠️ `양쪽불일치` | 매핑초과 + CSV초과 동시 발생 | 매핑과 CSV 구조 둘 다 확인 |
| 🔴 `전혀불일치` | matched 0개 → `value_vars=[]` → `metric_name KeyError` 원인 | 매핑 value_n 전면 수정 |
| ❌ `매핑없음` | 해당 tb가 매핑 파일에 없음 | 매핑 추가 또는 skip 여부 판단 |

## STEP 7. Post-Processing

기준 파일:

- `stack_n_currency_n_chnl_n_seaprate_날짜.ipynb`
- 참고 문서: `stack_n_currency_n_chnl_n_seaprate_날짜.md`

### 처리 순서 (260316 기준)

1. tb_key별 최신 파일 선택 (타임스탬프 기준, `_stacked` / `_long` / `union_` 제외)
2. Pre-scan: 전체 site_code 수집 (`us_` prefix → `_us_sites`, 그 외 → `_global_sites`)
3. 리포트 번호(`숫자_숫자`) 없는 파일 전체 skip (`bestselling`, `nextpage` 등)
4. `tb_column_name_mapping.csv` 기준으로 `value1~N` → 실제 컬럼명 rename
5. wide → long 변환 (`pd.melt`) 및 `currency.csv` 환율 적용 (revenue 컬럼만) → `metric_value_adj`
6. `J1~J7`, `J11(REPORT NO.)` 컬럼 분리
7. **non-channel 파일**: year-split 합산 (prior 기간이 연도 넘을 경우 — 시간 계열 평균, 나머지 합산) → dummy 0행 삽입 → `_stacked_separate.csv` 저장
8. **channel 파일**: US 채널명 → 글로벌 채널명 매핑 → `mc_needs_ch` / `mc_has_ch` 구분 → dummy 0행 삽입 (채널 × site 크로스곱) → PAID/NONPAID 부여 → `_stacked_separate.csv` 저장
9. union: non-US에 있고 US에 없는 **모든** metric_col에 US dummy 0행 자동 삽입 → `aa_exports/union_{YYYYMMDD}_{HHMMSS}.csv`

### channel 파일 mc 구분

| 구분 | 조건 | 처리 |
|------|------|------|
| `mc_needs_ch` | metric_col이 trailing `_`로 끝남 | value(채널명)를 metric_col 뒤에 붙여 KEY 생성 |
| `mc_has_ch` | metric_col에 채널명 내장 | value 그대로 유지, metric_col 변경 없음 |

> ⚠️ `tb_column_name_mapping.csv` 작성 시 채널 파일의 일반 revenue/order 컬럼은 반드시 trailing `_`로 끝나야 함

### 최종 출력 컬럼

```text
TIER, SUBS, COUNTRY, SITE CODE, REPORT NO., DIVISION, DATE, DEVICE TYPE,
TYPE, LOGIN/NON, PAID/NONPAID, ITEM, VALUE, KEY,
공란1, 공란2, 공란3, 공란4, value_origin, start_date, end_date
```

- `공란1~4`는 실제 레포트 양식에 맞춰 비워둔 4개 컬럼입니다. 이 외에 원본 값 참조용 `value_origin`과 추출 기간 확인용 `start_date`, `end_date` 컬럼이 추가로 포함되어 있습니다.
- `KEY = SITE CODE(소문자) + "_" + metric_col`
