---

## 목적

이 문서는 `260224_AA_Exporter(jiwoo)` 폴더에서 AA JSON 준비부터 노트북 생성, CSV 추출, 매핑 검수, 후처리 산출물 생성까지의 전체 흐름을 정리한 운영 문서입니다.

실제 기준 스크립트는 아래 4개입니다.

- `generate_period_notebooks_v2.py`
- `ipynb_json_usage_mapper.py`

- `stack_n_currency_n_chnl_n_seaprate_260306.ipynb`

## STEP 0. JSON Preparation

### 0-1. AA Workspace에서 1차 JSON 추출

- 기준은 당해 캠페인 JSON입니다.
- prior 기간과 세그 구성이 겹치는 테이블은 `26_ny_*_prior.json`으로 먼저 저장합니다.
- US는 별도 Report Suite를 쓰므로 NY와 분리해서 저장합니다.

### 0-2. prior JSON을 기준으로 non-prior / last campaign JSON 복사

`json/copy_prior_json.py`

- `26_ny_*_prior.json`을 `26_ny_*.json`으로 복사합니다.
- prior와 non-prior 구조가 같은 테이블의 초안 생성용입니다.

`json/copy_last_campaign_json.py`

- `26_ny_*.json`을 `25_ny_*.json`으로 복사합니다.
- 스크립트 상단 `FROM_YEAR = 26`을 기준으로 동작하므로 연도 전환 시 먼저 수정해야 합니다.
- 복사 후 `_cmp_`, `campaign`, `bestselling` 같은 캠페인 종속 세그는 25년용 세그 ID로 수동 교체가 필요합니다.

### 0-3. 빈 JSON 플레이스홀더 생성

`json/empty_json_maker_by_input_tb_name.py`

- prior에서 복사할 수 없는 테이블용 빈 JSON 파일을 일괄 생성합니다.
- 생성 직후에는 내용이 비어 있으므로 이후 AA에서 payload를 채워 넣어야 합니다.

### 0-4. AA Workspace에서 2차 JSON 마무리

- 비어 있는 JSON 파일을 차례로 채워 전체 테이블을 완료합니다.
- 추출 완료된 파일은 `-EMPTY` 접미사를 제거하면서 작업 상태를 정리합니다.

### 0-5. 미완성 JSON 표시

`json/mark_empty_json.py`

- 내용이 비어 있는 JSON에 `-EMPTY` 접미사를 다시 부여합니다.
- 최종 상태에서는 `-EMPTY` 파일이 0개가 되는 것이 정상입니다.
- 일괄 이름 복구가 필요하면 `json/rename_empty.py`를 사용합니다.

### 0-6. JSON 세그먼트 검수

`json/json_segment_checker.py`

결과 파일:

- `json_segment_report/_prior_check.csv`
- `json_segment_report/_26vs25_diff.csv`

검수 기준은 2가지입니다.

1. `26_ny_X.json` vs `26_ny_X_prior.json`
- prior와 완전히 같아야 정상인 테이블 확인용입니다.

2. `26_ny_X.json` vs `25_ny_X.json`
- `SHOULD_SAME`: 같아야 정상
- `SHOULD_DIFFER`: 달라야 정상
- 그 외: 참고용 표시

현재 문서 기준으로 특히 확인할 포인트는 아래입니다.

- 같아야 정상: `_scom_`, `6_1_`, `6_2_`, `4_2_`
- 달라야 정상: `_cmp_`, `campaign`, `3_1_`, `4_1_`, `bestselling`

이상 없으면 STEP 1로 진행합니다.

## STEP 1. Notebook Generation

실행 스크립트: `generate_period_notebooks_v2.py`

`json/` 폴더의 JSON 파일을 자동 감지해서 아래 6개 노트북을 생성합니다.

| 노트북 | 기간 | data_csv | JSON 패턴 |
|--------|------|----------|-----------|
| `01.26ny_campaign_period.ipynb` | 26 NY campaign | `26_ny_site_code.csv` | `26_ny_*.json` |
| `02.25ny_last_campaign_period.ipynb` | 25 NY last campaign | `25_bl_site_code.csv` | `25_ny_*.json` |
| `03.26ny_prior_period.ipynb` | 26 NY prior | `26_ny_site_code_prior.csv` | `26_ny_*_prior.json` |
| `04.us_26ny_campaign_period.ipynb` | 26 US campaign | `us_26_ny_site_code.csv` | `us_26_*.json` |
| `05.us_25ny_last_campaign_period.ipynb` | 25 US last campaign | `us_25_bl_site_code.csv` | `us_25_*.json` |
| `06.us_26ny_prior_period.ipynb` | 26 US prior | `us_26_ny_site_code_prior.csv` | `us_26_*_prior.json` |

실무 메모:

- 출력 파일명은 `OUT_NAMES`에서 바꿀 수 있습니다.
- 섹션 구분은 파일명 번호를 기준으로 자동 구성됩니다.
- `raw_multi_purchase`, `bestselling`, `nextpage`, `cc_order`, `cc_revenue`, `cc_visit`는 `base=""`로 처리됩니다.
- 각 코드 셀은 `utils.aa_exporter_jh2`를 사용하며 기본값은 `parallel_sites=True`, `max_workers=6`입니다.
- 후처리용 마지막 2개 셀은 `notebook_jh5-campaign_period.ipynb`에서 복사하도록 되어 있습니다. 해당 파일이 없으면 placeholder 셀이 들어갑니다.

## STEP 3. AA Data Extraction

생성된 6개 노트북을 순서대로 실행합니다.

- 출력 파일 형식: `aa_exports/{tb_key}_{YYYYMMDD}_{HHMMSS}.csv`
- 기본 컬럼 구조:
  `Subsidiary, Country, Site_Code, RSID, Start_Date, End_Date, itemId, value, value1, value2, ..., status, error`
- 병렬 추출: `parallel_sites=True`
- 기본 워커 수: `max_workers=6`
- 네트워크 오류 대응: 429, 5xx 계열 재시도 로직 포함

### 추출 후 FAILED 점검

실행 스크립트: `check_failed_status.py`

- `aa_exports` 루트의 원본 CSV만 검사합니다.
- `union*`, `*_stacked*`, `separate.csv`는 제외합니다.
- `status=FAILED` 건수가 있는 파일만 보여줍니다.

운영 기준:

- VRS, 일부 국가 RSID, 타임아웃 이슈는 수기 보정 대상이 될 수 있습니다.
- STEP 5 전에는 필요한 값이 모두 채워졌는지 반드시 확인합니다.

## STEP 4. Column Mapping Verification

실행 스크립트: `check_mapping_match.py`

결과 파일:

- `mapping_match_report/_all_check.csv`

검수 기준:

- 원본 CSV의 `value\d+` 컬럼과 `tb_column_name_mapping.csv`의 `value_n` 컬럼이 맞는지 확인
- 기본적으로 `aa_exports` 루트만 검사
- 날짜 보관 하위폴더(`260224`, `260226` 등)는 `SEARCH_SUBDIRS = False`라서 제외
- `union*`, `_stacked*`, `_long*`, `_`로 시작하는 파일도 제외

상태 해석은 아래 기준으로 보면 됩니다.

| 상태 | 의미 | 조치 |
|------|------|------|
| `정상` | 매핑과 CSV가 완전히 일치 | 그대로 진행 |
| `일부 매핑초과(CSV부족)` | 매핑에만 있고 CSV에는 없는 value 컬럼 존재 | 매핑 정리 또는 의도 확인 |
| `일부 CSV초과(매핑누락)` | CSV에만 있고 매핑에는 없는 value 컬럼 존재 | 매핑 행 추가 필요 |
| `일부 양쪽불일치` | 양쪽에 서로 다른 누락이 함께 존재 | 매핑과 CSV 구조 둘 다 확인 |
| `전혀 value불일치` | 매칭 0개 | 후처리 시 `value_vars=[]` 문제 원인 |
| `매핑없음` | 해당 tb가 매핑 파일에 없음 | 매핑 추가 또는 skip 여부 판단 |

## STEP 5. Post-Processing

기준 파일:

- `stack_n_currency_n_chnl_n_seaprate_260306.ipynb`
- 참고 문서: `stack_n_currency_n_chnl_n_seaprate_260306.md`

### 처리 순서

1. `tb_column_name_mapping.csv` 기준으로 `value1~N`을 실제 컬럼명으로 rename
2. wide 형식을 long 형식으로 변환
3. revenue 계열에 `currency.csv` 환율 반영
4. US 채널명을 글로벌 채널 기준으로 정리
5. 국가별 채널 수를 맞추기 위해 dummy 0행 추가
6. `J1~J7`, `J11(REPORT NO.)`를 분리
7. 개별 결과를 `aa_exports/{tb_key}_stacked_separate.csv`로 저장
8. 최종 통합본을 `aa_exports/union_{YYYYMMDD}_{HHMMSS}.csv`로 저장

### 현재 후처리 기준에서 반영된 중요 변경사항

- `value_vars`가 완전히 비는 경우 에러로 죽지 않고 skip 후 요약에 별도 표시
- `finalize_df()`를 거쳐 개별 파일과 union 모두 컬럼명/순서를 고정
- non-US에만 있는 `_app`, `_android`, `_ios` metric이 있으면 union 단계에서 US dummy 0행을 자동 삽입
- US 데이터가 전혀 없으면 `_app/_android/_ios` dummy insert는 warning만 출력하고 skip
- `ITEM` 컬럼, union 제외 대상 처리 로직이 보정된 상태

### 최종 출력 컬럼

```text
TIER, SUBS, COUNTRY, SITE CODE, REPORT NO., DIVISION, DATE, DEVICE TYPE,
TYPE, LOGIN/NON, PAID/NONPAID, ITEM, VALUE, KEY,
공란1, 공란2, 공란3, 공란4, value_origin, start_date, end_date
```

- `KEY = SITE CODE(소문자) + "_" + metric_col`
