## STEP 0. JSON Preparation (New Campaign Season)

캠페인이 새로 시작되거나 연도가 바뀔 때 JSON 쿼리 파일을 준비합니다.

### 0-1. 작년 캠페인용 JSON 복사
```
json/copy_last_campaign_json.py
```
- `26_ny_*.json` → `25_ny_*.json` 복사 (non-prior 파일만)
- 파일 상단 `FROM_YEAR` 값만 수정
- 복사 후 `25_ny_*_cmp_*`, `25_ny_*_bestselling*` 등 캠페인 세그 계열은
  수동으로 작년 세그 ID로 업데이트 필요

### 0-2. Prior JSON 복사 (필요 시)
```
json/copy_prior_json.py
```
- `26_ny_*_prior.json` → `26_ny_*.json` 복사
- 메인 파이프라인에서 prior 노트북은 `_prior.json` 직접 참조하므로 특수 상황 전용

### 0-3. 세그먼트 일관성 검수
```
json/json_segment_checker.py
→ json_segment_report/_prior_check.csv
→ json_segment_report/_26vs25_diff.csv
```
- 검사 1: `26_ny_X.json` vs `26_ny_X_prior.json` → 완전 동일해야 정상
- 검사 2: `26_ny_X.json` vs `25_ny_X.json`
  - **같아야정상** (`_scom_`, `6_1_`, `6_2_`, `4_2_`): ✅ 동일 / ❌ 다름(문제)
  - **달라야정상** (`_cmp_`, `campaign`, `3_1_`, `4_1_`, `bestselling`): ✅ 다름(업데이트됨) / ⚠️ 동일(25용 세그 미업데이트)
  - **특수케이스** (위 외): ℹ️ 결과만 표시
- 분류 기준은 스크립트 상단 `SHOULD_DIFFER` / `SHOULD_SAME` 목록에서 관리

### 0-4. 빈 JSON 관리 (필요 시)
```
json/empty_json_maker_by_input_tb_name.py  ← 이름 목록으로 빈 JSON 일괄 생성
json/mark_empty_json.py                    ← 내용 없는 JSON에 -EMPTY 접미사 붙임
json/rename_empty.py                       ← -EMPTY 접미사 제거 (실제 쿼리 붙인 후)
```

## STEP 3. AA Data Extraction

6개 노트북(`01~06`)을 순서대로 실행합니다.

- 출력: `aa_exports/{tb_key}_{YYYYMMDD}_{HHMMSS}.csv`
- 컬럼 구조: `Subsidiary, Country, Site_Code, RSID, Start_Date, End_Date, itemId, value, value1, value2, ..., status, error`
- 병렬 추출 (`parallel_sites=True`, `max_workers=6`) / 429·5xx 자동 retry

### 추출 후 FAILED 확인
```
check_failed_status.py
```
- CSV 파일별 FAILED 건수 확인
- VRS 등 일부 사이트는 FAILED → AA 수기 접속하여 값 직접 입력

## STEP 4. Column Mapping Verification

```
check_mapping_match.py
→ mapping_match_report/_all_check.csv
```
- `aa_exports/` 루트 CSV vs `tb_column_name_mapping.csv` 컬럼 매핑 검수
- 날짜 하위폴더(260224/ 등)는 검사 제외 (`SEARCH_SUBDIRS = False`)

| 상태 | 의미 | 조치 |
|------|------|------|
| ✅ 정상 | 완전 일치 | - |
| ⚠️ 매핑초과(CSV부족) | 매핑 value_n > CSV 컬럼 | 매핑 행 제거 또는 확인 |
| ⚠️ CSV초과(매핑누락) | CSV value19~20+ > 매핑 | 매핑에 추가 필요 |
| 🔴 전혀불일치 | 매칭 0개 → melt 오류 원인 | 매핑 value_n 전면 수정 |
| ❌ 매핑없음 | tb 자체 없음 | 매핑 추가 또는 skip 확인 |

## STEP 5. Post-Processing

```
stack_n_currency_n_chnl_n_seaprate_260306.ipynb
```
참고: `stack_n_currency_n_chnl_n_seaprate_260306.md`

### 처리 순서
1. `tb_column_name_mapping.csv` 기준으로 value1~N → 실제 컬럼명 rename
2. wide → long 변환 (value 값 1줄 1컬럼)
3. `currency.csv` 환율 적용 (revenue 컬럼만)
4. US 채널 → 글로벌 채널명 매핑 (US는 RS가 달라 채널명 다름)
5. 나라별 채널 수 통일을 위한 dummy 0행 삽입
6. J1~J7, J11(REPORT NO.) 컬럼 분리
7. 개별 파일 저장: `aa_exports/{tb_key}_stacked_separate.csv`
8. 전체 union: `aa_exports/union_{YYYYMMDD}_{HHMMSS}.csv` ← 최종 전달 파일

### 최종 출력 컬럼 (finalize_df 기준)
```
TIER, SUBS, COUNTRY, SITE CODE, REPORT NO., DIVISION, DATE, DEVICE TYPE,
TYPE, LOGIN/NON, PAID/NONPAID, ITEM, VALUE, KEY,
공란1, 공란2, 공란3, 공란4, value_origin, start_date, end_date
```
- KEY = SITE CODE(소문자) + "_" + metric_col
